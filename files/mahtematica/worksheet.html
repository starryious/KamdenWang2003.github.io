Mathematica isn't like most of the programming languages that many people know. And, at first, this can make it difficult to write structured, efficient Mathematica code. Hopefully this section will make some of the key differences clear.

There are several old families of programming languages that most modern languages are descended from. Almost all of the programming languages that are taught at school or university (to scientists at least), and most that are in everyday use are from the "Algol/Fortran" family. C, Pascal, Basic, Fortran, Java, C# are all from this family. Mathematica is not descended from Algol, but draws its core ideas from other language families (LISP, ML, rewrite systems). These ideas are very different to those behind the Algol family. To make matters more confusing, Wolfram (the makers of Mathematica) have tried very hard to make the system feel familiar to average users by presenting an Algol-like face. You might call Mathematica a rewrite system in LISP clothing, in Algol clothing. Or you might not. But either way keep in mind that when you're writing Mathematica code that the ideas behind the system might be different from what you're used to. Enough waffle - what are these ideas? Here are some of the key differences between Mathematica and the Algol family:

1) Uniform representation for everything. This is one of the most significant differences, but possibly the most abstract, so don't worry if it doesn't make too much sense now. Eventually it will. In the Algol family there are usually strong distinctions between source code, compiled object code, executable code, program data - these different types of data are usually stored in different formats and processed by different tools (the compiler, the linker, runtime system etc). Not so in Mathematica. Everything from the notebook files (the source code), to the functions in memory, to program data - graphics, sounds, tables, lists, vectors - are all represented by the same universal format and all processed by one tool, the Mathematica kernel. This has profound implications for the way code can be written. In particular, writing code that generates and modifies code, not data is very easy (data, code, who cares if it's all the same !?) and often very useful.

2) Functions are first class objects. (In fact, because of the uniform representation, everything can be and is a first class object). In most languages in common use there is a distinction between data (variables) and functions. And usually functions cannot be manipulated as easily or as thoroughly as variables. Hence the first-class (variables, data, objects in OO languages) and second-class (functions) monikers. In Mathematica you can assign functions to variables, write functions that take functions as parameters (functionals), operators that act on functions - anything you like really. [You can sort of do most of this in most Algol languages - using pointers in C, interfaces/anonymous inner classes in Java, delegates in C# etc - but it's often inelegant and quite limited. Mathematica really treats functions as just another kind of data (of course it does, because there's a uniform representation for everything!)]

3) List based. Everything in Mathematica is represented as an "expression". And these expressions are fundamentally list based (this is what Mathematica gets from LISP - the LIS in LISP stands for list). As a result it almost always pays to structure your code around lists and use higher order functions on these lists to achieve your aims. I'll explain what this means below as it's much easier to show by example.

4) Pattern matching and rules. The core of Mathematica is a powerful pattern-matcher. This is used behind the scenes to do an awful lot of things and it's also available for you to use. Rules allow you to use the pattern-matcher to effect transformation of expressions. They make it very easy to program some applications that are traditionally quite complex, but they do take a bit of getting used to.

5) Interactive, interpreted. In Algol-like languages one typically writes the code in a text editor, compiles the code to an executable and then runs the code, possibly with a debugger attached. This cycle is repeated over and over until the program does what you want. Not so in Mathematica: expressions are evaluated as soon as you enter them. This changes the way that you work - the program is built while it's running, debugging is done by writing code to inspect the running system, code just kind of grows by itself. It takes some discipline to write in this style - the main trick is that if something seems even slightly unsatisfactory then re-write it straight away - but it does allow you to be very productive once you get used to it.

But enough talk! Time to do some coding. Don't worry if the above stuff doesn't make a lot of sense right now (you might re-read it in a few weeks when you're on the way to becoming an expert). The main things to get used to to be productive in Mathematica are its list-orientation and the importance of defining functions. We'll go through some exercises now that hopefully illustrate what this is about.
